## PHASE 1 — TECHNICAL (27 elements)

Required data: sf-4xx.csv, sf-3xx.csv, sf-canonicals.csv, sf-mixed.csv, sf-urls.csv, sf-headers.csv, gsc-coverage.csv, psi-desktop.pdf, psi-mobile.pdf, gtmetrix.pdf, manual source code review

| # | Element | Description | Weight | Site-type notes |
|---|---|---|---|---|
| T01 | Indexation errors | GSC Coverage report: record total pages indexed vs not indexed. Categorise not-indexed reasons: Crawled not currently indexed (thin/duplicate content signal), Discovered not indexed (crawl budget issue), Duplicate without user-selected canonical (structural URL problem), Excluded by noindex, 404 errors, Blocked by robots.txt. Flag any significant spike in crawled-not-indexed vs prior period — often caused by a plugin, URL parameter, or CMS misconfiguration. Check mobile vs desktop parity. Admin pages, parameter URLs, and staging environments must not be indexed. | 10 | — |
| T02 | Bing indexation | Search site:domain.com in Bing to verify indexation and total page count. Access Bing Webmaster Tools: check sitemap submission, crawl errors, and blocked pages. Lower Bing index count than Google is normal if Google over-indexes duplicate pages. Flag if site is completely absent from Bing results — indicates a WMT configuration issue or robots.txt blocking Bingbot. | 5 | Higher weight for B2B/SaaS |
| T03 | Sitemap.xml | Verify sitemap present (domain.com/sitemap.xml), submitted and accepted in GSC with no errors. Check: (1) no 3xx redirected URLs — only 200-response pages; (2) no 404 URLs; (3) no noindex pages listed; (4) lastmod tag present on all entries; (5) image attributes per entry point to actual page/product image — not a generic logo placeholder; (6) all major site sections navigable from homepage are included; (7) parameterised/tracking URLs excluded; (8) old deleted pages removed. Flag total URLs in sitemap vs total indexed in GSC. If URLs need to be added or removed from the sitemap, write a separate sitemap-delta.csv file with two columns: URLs to Add, URLs to Remove. | 10 | — |
| T04 | Robots.txt | Verify robots.txt present and syntactically valid. Check: (1) no strategic pages accidentally blocked; (2) URL parameters and tracking strings disallowed; (3) admin, staging, and plugin-generated URLs disallowed; (4) Sitemap directive declared; (5) AI bots not blocked — check for GPTBot, ClaudeBot, PerplexityBot, Google-Extended; (6) no overly broad Disallow: / rule. Write a robots-txt-recommendation.txt file with: section 1 — current robots.txt content; section 2 — recommended robots.txt with each change annotated with a reason. Link this file in the Sources column. | 10 | — |
| T05 | Canonical tags | Primary source: GSC Coverage report — filter by 'Alternate page with proper canonical tag' and 'Duplicate without user-selected canonical' states to identify canonical issues at scale. Secondary source: sf-canonicals.csv for row-level URL inspection. Cross-reference both — GSC reveals canonicalization issues on pages SF may not have crawled (parameterized URLs from filter interactions, ad click IDs, UTM parameters, and pagination are not crawled by SF but appear in GSC). Every indexable page must have a self-referential canonical. Flag: (1) canonical pointing to a different URL — distinguish intentional consolidation from misconfiguration; (2) multiple conflicting canonical tags; (3) canonical pointing to a 3xx redirect; (4) canonical pointing to a 404; (5) pages with no canonical tag. Write canonical issues to canonical-tag-issues.csv. If GSC is unavailable: score from SF data alone and explicitly note in the Score Explanation that parameterized URLs (filter params, ad click IDs, tracking params, pagination) are not visible to SF — the canonical issue count may be understated. Flag as partial and rescore when GSC becomes available. | 10 | — |
| T06 | 404 errors | Primary source: GSC Coverage report — filter by 'Not found (404)' status for all Google-discovered 404s, including URLs that appear in GSC but were never crawled by SF. Secondary source: sf-4xx.csv for crawl-discovered 404s with link context (internal/external). Always cross-reference both — GSC reveals 404s on low-priority-crawl URLs that Screaming Frog misses (JS-generated paths, old ad landing URLs, parameterized variants, historical links). All discovered 404 URLs must be listed in the redirect-plan.csv and in the Score Explanation, regardless of whether they are redirect candidates — annotate each with its action: 301 redirect, clean up, block in robots.txt, or no action required. Never omit URLs because they appear low priority — full disclosure prevents the audit from appearing to have missed data. Triage by action type: (1) 404s with inbound internal links → redirect immediately; (2) 404s in sitemap → remove and redirect; (3) 404s with inbound external backlinks → highest link equity loss; (4) parameterized/JS-generated paths → add canonical to clean URL or block in robots.txt; (5) GTM/GA misconfiguration paths → GTM audit required, do not redirect; (6) 0-inbound-link orphans → clean up or leave. If GSC is unavailable: score from SF data alone and explicitly note in the Score Explanation that the 404 count is likely understated — SF only discovers URLs reachable through the current internal link graph, while GSC captures URLs from external backlinks, historical crawls, ad click traffic, and JS-generated navigation events. Flag as partial and rescore when GSC becomes available. | 7.5 | — |
| T07 | Redirect chains | Source: sf-3xx.csv. A redirect chain is Page A redirecting to Page B redirecting to Page C. All chains must be collapsed to a direct 301 to the final 200-response URL. Audit redirect types: 302 redirects must not be used in place of 301s for moved or consolidated pages. Report total chain count and maximum chain depth found. | 7.5 | — |
| T08 | HTTPS/WWW consistency | Source: sf-mixed.csv. All four domain variants must 301-redirect to a single canonical version. Mixed content — HTTP resources loaded on HTTPS pages — generates browser security warnings. Verify in source code and Screaming Frog Mixed Content export. | 10 | — |
| T09 | SSL certificate | PROMPT USER. Verify SSL is valid, active, and covers all subdomains in use. Flag if expiring within 60 days — auto-renewal should be confirmed. An expired SSL triggers browser security warnings, spikes bounce rate, and can reduce rich result eligibility. | 10 | — |
| T10 | Secured login / admin | PROMPT USER. The CMS admin panel must not be accessible via the default URL. Check: (1) admin URL changed to a non-default path; (2) two-factor authentication enabled; (3) login attempt rate limiting. Also check that security plugins are not generating publicly crawlable parameterised URLs. | 5 | Higher weight for WordPress; Not Applicable for fully-managed platforms |
| T11 | CMS plugins & platform version | PROMPT USER. CMS core and all active plugins must be current. Outdated versions expose security vulnerabilities and can generate crawlability issues. Check GSC Crawl Stats: if Googlebot is spending a disproportionate share of crawl time loading page resources vs crawling content pages, plugin bloat is the likely cause. | 5 | Higher weight for WordPress; Not Applicable for Shopify/managed platforms |
| T12 | External linking noopener | PROMPT USER. All external links with target="_blank" must include rel="noopener noreferrer". Without noopener, the newly opened tab can access the opener window's object — a security vulnerability known as reverse tabnapping. | 2.5 | — |
| T13 | reCAPTCHA / anti-spam | PROMPT USER. Bot protection must be active on all public-facing forms: contact, newsletter signup, checkout, account creation, and comment sections. Comment sections not managed through a trusted third party carry the highest spam risk. | 2.5 | Higher weight for lead-gen sites, e-commerce, and any site with open comment sections |
| T14 | PageSpeed score — desktop | Source: psi-desktop.json. Report: Performance score (/100), LCP (pass: 2.5s or under), TBT (pass: 200ms or under), CLS (pass: 0.1 or under), FCP. In the How to Correct field, parse psi-desktop.json for every audit entry where score < 1 (opportunities and diagnostics with scoreDisplayMode not equal to 'notApplicable' or 'manual'). List each item in descending impact order using this format: '(N) [Title] — [estimated savings or metric value] — [one-sentence action]'. Do not summarise — every flagged item must appear individually. Add the live PageSpeed Insights URL to the Sources column: https://pagespeed.web.dev/report?url=[percent-encoded domain URL]&strategy=desktop | 7.5 | — |
| T15 | PageSpeed score — mobile | Source: psi-mobile.json. Mobile PSI is the primary signal for Google Mobile-First Indexing. Same How to Correct format as T14 — parse psi-mobile.json and list every flagged item individually. Mobile-specific critical thresholds: LCP above 4s is critical; Time to First Byte above 600ms indicates a server response issue; any navigation elements relying on JavaScript to render are not crawlable by Googlebot. Add the live PageSpeed Insights URL to the Sources column: https://pagespeed.web.dev/report?url=[percent-encoded domain URL]&strategy=mobile | 10 | — |
| T16 | GTmetrix score | Source: gtmetrix.pdf. Report: Grade (A–F), Performance %, Structure %, LCP, TBT, Total Page Size, Number of Requests. Key checks: total page weight — flag if above 3MB; CDN usage for static assets; JavaScript execution time; waterfall view — identify the single slowest-loading asset. | 7.5 | — |
| T17 | JS & CSS minification | Unused JavaScript and CSS inflate page weight and extend render-blocking time. Identify: (1) total KB of unused JS; (2) total KB of unused CSS; (3) legacy scripts from removed features still loading; (4) third-party scripts loading synchronously; (5) theme/plugin stylesheets loading globally when only needed on specific page types. | 7.5 | — |
| T18 | Lazy load | Lazy loading defers loading of below-fold images and iframes until the user scrolls near them. Verify: loading="lazy" attribute on img tags, or JavaScript Intersection Observer implementation. | 5 | — |
| T19 | CDN (images & server) | Source: sf-headers.csv. A Content Delivery Network serves assets from servers geographically close to the user, reducing latency. Check: (1) images served via CDN — check response headers; (2) JS/CSS static assets served via CDN; (3) origin server geolocation — server should be in or near the primary user geography. Do not recommend a specific CDN provider. Describe what to implement, not which product to use. | 7.5 | Higher weight for international or multi-region sites |
| T20 | HTML/text ratio | PROMPT USER. The ratio of visible text to total HTML on a page. Target: 20% or above. A ratio below 5–10% signals a code-heavy, content-thin page. Common causes: excessive nested divs, large inline JS/CSS, very short product descriptions. Fix by adding substantive content — not by removing HTML. | 2.5 | — |
| T21 | W3C HTML validation | PROMPT USER. Run key page templates through validator.w3.org. Common errors: img elements missing alt attributes; invalid characters in href attributes; stray/unclosed tags; heading elements out of sequential order. | 5 | — |
| T22 | UTF-8 encoding | Verify meta charset="UTF-8" present in the head of all pages. UTF-8 encoding ensures correct rendering of all characters across all browsers and prevents character encoding errors that corrupt visible content and confuse parsers. | 2.5 | — |
| T23 | Server type & hosting | PROMPT USER. Identify server technology and hosting model (shared / VPS / dedicated / cloud). Check: (1) how many other sites share the same IP address via DomainTools; (2) hosting model appropriateness for site traffic volume; (3) whether server location matches the primary user geography. Flag if a large or high-traffic site is on shared hosting. | 5 | Higher weight for larger sites or sites with significant traffic |
| T24 | SameAs tag (schema) | PROMPT USER. The sameAs property in Organization schema explicitly connects the website to its verified social media profiles, Wikipedia page, and other authoritative external sources. Check: Organization entity present on homepage; sameAs array populated with all active profiles (Facebook, Instagram, TikTok, X/Twitter, YouTube, LinkedIn, Pinterest as applicable); all sameAs URLs are live and correctly point to official brand accounts. | 10 | — |
| T25 | JavaScript — important elements | Key content and navigation elements must not depend on JavaScript to render. Check: (1) primary navigation links are present in the HTML source and do not require JS execution to appear; (2) product names, descriptions, and prices are in the HTML — not injected via JS after page load; (3) internal links to category and product pages are crawlable without JS; (4) test by disabling JavaScript in the browser and confirming the page's primary content and navigation remain functional. Googlebot can execute some JavaScript but crawls it on a delayed basis — JS-dependent content may not be indexed promptly or at all. | 7.5 | Higher weight for e-commerce and JS-heavy frameworks (React, Angular, Vue) |
| T26 | Star rating system (aggregate rating) | PROMPT USER first (see PROMPT USER table) — ask whether the site has a visible star rating or customer review display before doing any code inspection. Score based on the owner's answer: if Not Present, score immediately without code inspection. Only inspect code if the owner confirms a rating system is present. An on-site star rating or review system enables AggregateRating schema, which makes star ratings eligible to appear directly in Google search results (rich results). If present, check: (1) product or service pages display a visible star rating or review module; (2) AggregateRating schema is implemented with ratingValue, reviewCount, and bestRating properties; (3) ratings are genuine user-generated scores — fabricated ratings violate Google's guidelines and can result in manual action; (4) run key pages through the Rich Results Test to confirm star rating eligibility. How to Correct must describe what to implement, not which platform or product to use. | 5 | Higher weight for e-commerce, local business, and SaaS with user reviews |
| T27 | Schema markup — implementation & coverage | Source: sf-schema.csv. Run all key page templates through the Google Rich Results Test. This element scores overall schema health — the Schema Sub-Phase provides the full per-page-type deep-dive. Check: (1) is any schema markup present sitewide; (2) is JSON-LD used (preferred over microdata or RDFa); (3) are page-specific schema types implemented where supported — Product on product pages, Recipe on recipe pages, Organization on homepage, AboutPage and ContactPage where applicable, BreadcrumbList on all inner pages; (4) are there critical errors blocking rich result eligibility in the Rich Results Test; (5) is sameAs present on the Organization entity linking to all active social profiles. Note: T24 (SameAs) and T26 (Star Rating / AggregateRating) findings directly compound this score — score all three before writing the phase summary. | 10 | Weight applies universally — schema gaps affect all site types. Schema Sub-Phase (Mode C) provides the full implementation specification per page type. |

## PHASE 2 — USER EXPERIENCE (14 elements)

Required data: sf-internal-links.csv, psi-mobile.pdf, gsc-mobile.csv, manual site review

| # | Element | Description | Weight | Site-type notes |
|---|---|---|---|---|
| U01 | Mobile responsiveness | PROMPT USER. The site must be fully responsive across screen sizes. Check: (1) no horizontal scrolling; (2) all buttons and CTAs tappable without zoom; (3) no content clipped or overflowing at mobile viewport; (4) navigation accessible on mobile; (5) mobile LCP should not exceed 4s. Google Mobile-First Indexing means the mobile version is what Google indexes and ranks. | 10 | — |
| U02 | Core Web Vitals | Source: gsc-mobile.csv. Report status per metric: LCP (pass: 2.5s or under), CLS (pass: 0.1 or under), INP (pass: 200ms or under). Distinguish lab data (PSI) from field data (GSC) — GSC field data is what Google uses for ranking. Flag any metric showing Poor status for a significant percentage of pages. | 10 | — |
| U03 | Navigation depth / crawl depth | Every important page should be reachable within 3 clicks from the homepage. Check: (1) key pages are not orphaned — they must be linked from navigable pages; (2) no pages that are indexed and canonical but not accessible through normal navigation; (3) footer and header navigation covers all strategic sections. | 7.5 | — |
| U04 | Internal linking structure | Source: sf-internal-links.csv. Check: (1) total internal link count per page; (2) anchor text is semantically descriptive — not generic; (3) no dead-end pages; (4) no broken internal links; (5) key pages receive sufficient internal link equity; (6) link equity flows logically from homepage to categories to products. | 7.5 | — |
| U05 | Breadcrumbs | Check: (1) breadcrumbs present on all pages below homepage level; (2) breadcrumb trail reflects the actual URL hierarchy; (3) each breadcrumb segment uses the keyword-optimised page name; (4) BreadcrumbList schema implemented with item URLs on every ListItem; (5) no navigation bugs causing breadcrumbs to display incorrectly. | 5 | — |
| U06 | 404 page quality | PROMPT USER. Check: (1) custom 404 page exists; (2) includes site navigation, search bar, or links to popular pages; (3) clearly communicates the error in accessible language; (4) returns correct HTTP 404 status code — not 200 (a soft 404 is worse than a hard 404). | 2.5 | — |
| U07 | Footer configuration | PROMPT USER. Check: (1) all major site sections linked in footer; (2) no strategic sections missing from both header and footer; (3) links are crawlable and not JS-dependent; (4) footer includes trust signals (privacy policy, terms, copyright). | 5 | — |
| U08 | Above-the-fold content | PROMPT USER. Check: (1) primary H1 visible above the fold; (2) CTA accessible without scrolling; (3) no interstitials or large cookie banners blocking primary content on first load; (4) hero image or primary visual relevant to page content. | 5 | — |
| U09 | Interstitials / pop-up compliance | PROMPT USER. Google penalises intrusive interstitials on mobile that cover the main content shortly after page load. Penalisable: full-screen popups that must be dismissed before content is visible; large banners covering main content. Acceptable: cookie consent banners, small banners not covering main content, popups triggered by exit intent or after significant scroll. | 7.5 | — |
| U10 | Font legibility / tap targets | PROMPT USER. Minimum body font size: 16px. Tap targets should be 48x48px or larger. Check: (1) no text below 16px for body content; (2) tap target size and spacing on mobile; (3) sufficient colour contrast (WCAG AA: 4.5:1 for normal text). | 5 | — |
| U11 | Social sharing features | PROMPT USER. Check: (1) social share buttons present on blog articles, recipes, and product pages; (2) all major platforms represented; (3) share functionality works correctly on mobile; (4) OpenGraph tags correctly populated so shared links generate rich previews. | 2.5 | Higher weight for media, recipe, and blog-heavy sites |
| U12 | Contact accessibility | PROMPT USER. Check: (1) contact form present on dedicated Contact page; (2) contact link in main navigation or footer; (3) phone number clickable on mobile; (4) email address present and mailto: linked. Contact accessibility is also an E-A-T trust signal. | 5 | Higher for service businesses; Not Applicable for pure content/media sites |
| U13 | Favicon | PROMPT USER. Check: (1) favicon present and correctly configured; (2) favicon is legible at small sizes; (3) Apple touch icon also configured. | 2.5 | — |
| U14 | Newsletter / email capture | PROMPT USER. An email capture mechanism (newsletter signup, discount offer, lead magnet) is a conversion and retention element. Check: (1) newsletter or email signup option is present and accessible — in the footer, a dedicated landing page, or as a non-intrusive inline element; (2) signup form is functional and bot-protected; (3) confirmation or thank-you message displays after submission; (4) the signup is not implemented as an intrusive interstitial that triggers on page load (which would also be flagged under U09). | 2.5 | Higher weight for e-commerce, media, and content-heavy sites; Not Applicable for pure B2B service or local business sites where email capture is not a primary conversion goal |

| C01 | Google Search Console setup | GSC should be verified for all domain variants. Check: (1) correct property type (Domain property preferred); (2) sitemap submitted and showing no errors; (3) primary domain verified; (4) GSC linked to GA4. Without GSC, indexation issues, manual penalties, and coverage errors are invisible. | 10 | — |
| C02 | Google Analytics / GA4 setup | Source: ga4-events.csv. Check: (1) tracking code present on all pages; (2) no duplicate tracking; (3) key events configured; (4) data retention set to maximum (14 months); (5) GA4 linked to GSC; (6) e-commerce tracking enabled if applicable; (7) Google Ads linked if running paid campaigns. | 10 | — |
| C03 | Bing Webmaster Tools | PROMPT USER. Bing WMT provides Bing-specific crawl data and is essential for LLM visibility since ChatGPT and Copilot use Bing as their primary search data source. | C04 | Google Tag Manager | PROMPT USER. GTM centralises all tracking and marketing tags. Check: (1) GTM container present and firing on all pages; (2) no redundant or conflicting tags; (3) no broken or misconfigured triggers; (4) remarketing tags present if running paid campaigns. | 5 | — |
| C05 | Domain renewal date | Source: whois.txt. Check domain expiry date. Flag if expiring within 90 days — recommend immediate renewal with auto-renewal enabled. An expired domain goes offline completely. | 7.5 | — |
| C06 | WHOIS / hosting info | Source: whois.txt. Check: (1) registrar; (2) hosting type and provider; (3) IP address and how many other sites share it; (4) WHOIS privacy protection active. | 2.5 | — |
| C07 | Hreflang / international config | PROMPT USER. | C10 | Traffic source breakdown | Source: ga4-traffic.csv. Report: (1) organic search as percentage of total sessions; (2) search engine split within organic; (3) LLM referral traffic — check referral sources for chatgpt.com, perplexity.ai, gemini.google.com, you.com, bing.com/chat, claude.ai; (4) channel trend MoM or YoY. | 7.5 | [EVOLVING] — LLM referral sources expanding monthly |
| C11 | Remarketing & pixel configuration | PROMPT USER. Check: (1) Google Ads remarketing tag present if running Google Ads; (2) Meta Pixel present if running Facebook/Instagram Ads; (3) no duplicate pixel firing; (4) pixel events correctly configured. | 5 | Higher weight for e-commerce and lead-gen; Not Applicable for pure content/informational sites |

## PROMPT USER — Phases 1, 2, 3

For each element below, ask the **exact question listed** before scoring — verbatim, without paraphrasing or summarising. Ask all PROMPT USER questions for a phase in one batch at the start of that phase — do not interrupt element-by-element.

If the user cannot answer a PROMPT USER question, score that element as DATA MISSING and note: "Unable to verify — manual check required before finalising score."

**Writing rule — neutral language**: When recording answers in Data Analyzed or Score Explanation, do not write "the user confirmed," "user-provided," or "user said." Write the finding as a fact or use neutral language: "verified," "confirmed by site owner," "manually verified." This is a client-facing deliverable — automated-audit language must not appear.
